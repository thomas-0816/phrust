# Performance Acceleration Known Gaps

This catalog tracks the gaps that block the staged acceleration program from
being considered complete. It does not replace `docs/performance-known-gaps.md`
or `docs/performance-cranelift-known-gaps.md`; it coordinates the next
interpreter, bytecode, quickening, inline-cache, runtime-fast-path, release, and
default-off native-tier work.

Generated reports under `target/performance/` are local evidence only and must
not be committed.

## Entry Format

- `Gap ID`: stable acceleration identifier used by docs, reports, and final
  audits.
- `Layer`: engine area blocked by the gap.
- `Current evidence`: current file, command, or absence proving the gap.
- `Risk`: correctness, safety, measurement, or policy issue.
- `Planned handoff`: prompt-pack phase expected to close or refine the gap.

## Closure Policy

A gap is closed only when the owning phase lands code or docs with all required
off switches, fallback paths, counters, fixtures, reports, and validation
evidence. Wall-clock-only evidence cannot close an acceleration gap.

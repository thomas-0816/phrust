# Error Recovery

The parser must preserve source text while recovering from malformed input.

## Recovery Sets

Recovery boundaries are centralized in `crates/php_syntax/src/recovery.rs`.
Grammar modules may use narrower local lookahead for language-specific starts,
but synchronization should fall back to these shared sets:

- `STMT_RECOVERY`: `;`, `}`, `T_CLOSE_TAG`, EOF
- `EXPR_RECOVERY`: `,`, `;`, `)`, `]`, `}`, `T_CLOSE_TAG`, EOF
- `PARAM_RECOVERY`: `,`, `)`, EOF
- `ARG_RECOVERY`: `,`, `)`, EOF
- `MEMBER_RECOVERY`: `;`, `}`, EOF
- `TYPE_RECOVERY`: `,`, `)`, `;`, `=`, `{`, EOF
- `ATTRIBUTE_RECOVERY`: `]`, `)`, EOF

Statement parsing synchronizes at `STMT_RECOVERY`. Expression parsing
synchronizes at `EXPR_RECOVERY`, with grammar-specific extra stops such as
`echo` where that token begins a new construct in the current context.

PHP close tags are valid statement terminators. The statement parser does not
consume `T_CLOSE_TAG`; the surrounding PHP block parser owns that token so tags
remain visible in the CST.

All shared recovery loops route through a progress guard. Each recovery step
must either consume a token or reach EOF; debug builds assert that invariant and
release builds break rather than spin if a future boundary predicate is wrong.

## Diagnostic IDs

Parser diagnostics use stable snake-case IDs:

- `unexpected_token`
- `expected_token`
- `expected_expression`
- `expected_statement`
- `expected_type`
- `expected_identifier`
- `unclosed_delimiter`

Lexer diagnostics forwarded through the parser use `lexer_diagnostic`.

Each parser diagnostic carries:

- the stable ID,
- a byte `span`,
- an `expected` syntax list,
- severity `error`.

Expected-syntax diagnostics are classified centrally from the expected syntax
set. Grammar code should pass specific expected names such as `expression`,
`type`, `T_STRING`, `;`, or `}` so diagnostics stay stable and useful.

## Error Nodes

Recovery diagnostics emit `ERROR` CST nodes at the diagnostic span. When a
grammar production consumes malformed input during recovery, the surrounding
recovery marker is also completed as an `ERROR` node, so the bad token range is
preserved in the tree. Source reconstruction must still exactly match the input.

## Tests

Recovery coverage includes:

- parser unit tests for stable diagnostic IDs and expected syntax,
- parser unit tests that malformed byte-oriented smoke inputs terminate and
  roundtrip,
- `just cst-roundtrip` for exact source reconstruction across all committed
  parser fixtures,
- deterministic parser property tests in
  `crates/php_syntax/tests/parser_properties.rs`,
- recovery fixtures under `fixtures/parser/recovery/` for missing expressions,
  unclosed delimiters, bad attributes, and PHP block boundaries.

Reference-dependent fixture commands may report these recovery fixtures as
rejected by PHP where they are intentionally malformed. The Rust parser should
emit diagnostics and a lossless CST rather than panic or hang.

The property smoke checks a bounded set of hand-written malformed inputs and
deterministically generated byte-derived strings. For each case it verifies:

- parsing does not panic,
- parsing terminates within the per-case timeout,
- `reconstructed_text() == source`,
- diagnostic spans stay within the source byte range,
- CST element spans stay within the source byte range,
- the root range stays within the source byte range.

The normal workspace test run keeps this smoke short. A longer local run is
available with:

```bash
nix develop -c just fuzz-parser-smoke
```

The long smoke is still a bounded regression check, not coverage-guided fuzzing.
When it finds a panic or nontermination case, reduce that input into a committed
fixture or unit test before changing parser behavior.
